# Рекомендации по устранению узких мест

## Высокий приоритет
| Мера                                                                     | Обоснование                                                                     |
| ------------------------------------------------------------------------ | ------------------------------------------------------------------------------- |
| 1. Внедрить шифрование ПДн и медицинских данных (at rest и in transit)   | Критически важно для соответствия ФЗ-152 и предотвращения утечек.               |
| 2. Настроить регулярные резервные копии и тестировать восстановление     | Потеря данных — угроза существованию системы. Без бэкапов — нет восстановления. |
| 3. Внедрить аудит всех действий с персональными и медицинскими данными   | Необходимо для контроля доступа, выявления инцидентов и отчётности.             |
| 4. Внедрить DLP-систему для контроля выгрузок (Excel, email, USB)        | Предотвращает утечки через файловые выгрузки — один из главных векторов риска.  |
| 5. Утвердить политику удаления персональных данных (по запросу и срокам) | Требование ФЗ-152. Отсутствие политики — юридический риск.                      |

## Средний приоритет
| Мера                                                              | Обоснование                                                                            |
| ----------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| 6. Перейти на Kubernetes для оркестрации и стандартизации деплоев | Устраняет ручные ошибки, повышает надёжность и масштабируемость.                       |
| 7. Внедрить CI/CD (сборка, тесты, security-сканы, деплой)         | Снижает время выхода на продакшн, повышает качество и безопасность.                    |
| 8. Заменить Excel на PostgreSQL и единые API для доступа к данным | Устраняет дубли, ошибки и несогласованность. Основа для интеграций.                    |
| 9. Нанять фронтенд-разработчика (JS/React)                        | Необходимо для создания современного портала пациента и админки.                       |
| 10. Ввести классификацию данных (PII/PHI/операционные)            | Позволяет применять разные уровни защиты и контроля в зависимости от чувствительности. |

## Низкий приоритет
| Мера                                                                  | Обоснование                                                             |
| --------------------------------------------------------------------- | ----------------------------------------------------------------------- |
| 11. Автоматизировать запись пациентов и уведомления                   | Повышает удобство, но не угрожает стабильности или безопасности.        |
| 12. Организовать обучение персонала по безопасности и новым процессам | Долгосрочная мера, но без неё возможны срывы при внедрении.             |
| 13. Подготовить и донести план отката релизов                         | Важно для уверенности команды, но актуально после внедрения CI/CD.      |
| 14. Собрать обратную связь от сотрудников по миграции                 | Помогает снизить сопротивление изменениям, но не технический приоритет. |

# Итоговая оценка приоритетов
| Категория                      | Приоритет | Обоснование                            |
| ------------------------------ | --------- | -------------------------------------- |
| Безопасность и соответствие    | Высокий   | Риски: утечки, штрафы, потеря лицензии |
| Надёжность и восстановление    | Высокий   | Риски: потеря данных, простои          |
| Инфраструктура и автоматизация | Средний   | Риски: ошибки, медленные релизы        |
| Данные и интеграции            | Средний   | Риски: несогласованность, дубли        |
| Персонал и процессы            | Низкий    | Риски: сопротивление, задержки         |
