# 1. Сравнение текущих практик с требованиями безопасности
| Область безопасности  | Текущее состояние (As-Is)                                                                | Требования (To-Be / Best Practices)                                                               | Соответствие?   |
| --------------------- | ---------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- | --------------- |
| Хранение данных       | Данные хранятся в Excel, PDF, JPG, 1С, на файловом сервере без шифрования.               | Данные должны шифроваться при хранении (Encryption at Rest), особенно персональные и медицинские. | ❌ Нет           |
| Передача данных       | Нет явного шифрования при передаче между системами (OLE, TCP/IP).                        | Должно использоваться шифрование в движении (TLS 1.2+).                                           | ❌ Нет           |
| Контроль доступа      | Доступ к файлам - по доменной аутентификации, без RBAC/ABAC. Нет аудита прав.            | Должна быть реализована модель управления доступом (RBAC/ABAC), основанная на ролях и атрибутах.  | ❌ Нет           |
| Аудит действий        | Отсутствует ведение журналов доступа к данным.                                           | Должны вестись логи всех критических операций: доступ, изменение, удаление.                       | ❌ Нет           |
| Целостность данных    | Нет контроля изменений: файлы могут редактироваться без отслеживания.                    | Должны быть механизмы контроля целостности (хеши, версионирование).                               | ❌ Нет           |
| Минимизация данных    | Собирается избыточная информация (адрес, место работы), включая хронические заболевания. | Должна собираться только та информация, которая необходима для оказания услуги.                   | ❌ Нет           |
| Согласие на обработку | Нет системы управления согласиями пациентов.                                             | Должна быть возможность получения, хранения и отзыва согласия.                                    | ❌ Нет           |
| Удаление данных       | Нет политики удаления данных после окончания срока хранения.                             | Должны быть механизмы автоматического удаления или архивации по истечении срока.                  | ❌ Нет           |
| Локализация данных    | Сервер находится в офисе - соответствует 152-ФЗ.                                         | Требуется подтверждение, что данные не реплицируются за пределы РФ.                               | ✅ Да (частично) |
| Резервное копирование | Есть файловый сервер, но неясно, настроено ли резервное копирование.                     | Должны быть регулярные, зашифрованные резервные копии с проверкой восстановления.                 | ⚠️ Неясно        |
| Обработка PII/PHI     | Персональные и медицинские данные хранятся в открытом виде.                              | Должна быть классификация данных, тегирование, применение мер защиты (шифрование, маскировка).    | ❌ Нет           |

# 2. Проблемные зоны - список рисков
Ниже представлен перечень критических проблем безопасности, выявленных в текущей системе:

## 1. Хранение персональных и медицинских данных без шифрования
- Данные в Excel, PDF и 1С доступны любому сотруднику с доступом к файловому серверу.
- Риск: утечка при краже устройства, увольнении сотрудника, взломе.

## 2. Отсутствие контроля доступа на уровне данных
- Любой сотрудник ресепшена может открыть любую папку пациента.
- Нарушение принципа минимальных привилегий.

## 3. Нет аудита доступа к данным
- Невозможно отследить, кто и когда просматривал или изменял данные пациента.
- Проблема при расследовании утечек или внутренних нарушений.

## 4. Сбор избыточной информации без согласия
- Пациенты указывают данные, не связанные напрямую с оказанием услуги (место работы, хронические заболевания).
- Несоответствие принципу Data Minimization и ФЗ-152.

## 5. Ручная обработка данных
- Все процессы (учёт, запись, платежи) - ручные, высок риск ошибок и несанкционированного копирования.

## 6. Отсутствие управления согласиями
- Нет возможности доказать, что пациент давал согласие на обработку данных.

## 7. Нет политики жизненного цикла данных
- Неясно, как и когда удаляются данные пациентов.
- Риск хранения данных дольше необходимого.

## 8. Интеграция 1С через OLE/TCP/IP без шифрования
- Передача данных между ККМ и 1С - открытым способом.
- Возможна перехват информации о платежах.

## 9. Файловый режим 1С
- Ограниченная масштабируемость, высокий риск повреждения базы, нет централизованного доступа.

## 10. Отсутствие DLP-систем
- Нет автоматического обнаружения утечек (например, отправка файлов на email).

## 11. Нет тегирования данных
- Невозможно автоматически определить, какие файлы содержат PII/PHI.
